- [ ] 配置 GitHub Actions：多 Python 版本测试、lint、类型检查和构建。

## 2. 领域模型与工作流状态机（M1）

- [ ] 定义 `Project`：任务说明、论文类型、篇幅、截止时间、学校模板、引用样式、输出要求。
- [ ] 定义 `Document`、`DocumentChunk`、页码/段落定位和解析警告。
- [ ] 定义 `SourceRecord`、`EvidenceCard`、`Claim`、`OutlineNode`、`DraftSection`。
- [ ] 定义 `WritingSkill` Schema：规则、例证、支持度、冲突、置信度、版本和验证结果。
- [ ] 定义 `AuditFinding`：严重度、位置、证据、修复建议和处理状态。
- [ ] 实现可恢复的工作流状态机，阶段输出写入本地项目目录。
- [ ] 为每次运行记录配置快照、模型标识、Prompt/Skill 版本和输入文件哈希。
- [ ] 设计向后兼容的 Schema 迁移机制。

验收：中途退出后可以从上一个已完成阶段继续；同一输入和固定模型响应可重放；所有结论能追溯到输入与版本。

## 3. 多模型适配层（M1）

- [ ] 定义供应商无关的 `ModelClient` 接口：文本生成、结构化输出、流式输出、工具调用和 Embedding（可选）。
- [ ] 通过配置文件和 CLI 参数选择 provider/model，不在业务代码内判断供应商。
- [ ] 首批支持至少两个云端供应商和一个 OpenAI-compatible/local endpoint。
- [ ] 实现能力探测：上下文长度、结构化输出、工具调用、温度支持情况。
- [ ] 实现超时、重试、速率限制、并发控制和指数退避。
- [ ] 实现 Token/费用统计及每阶段预算上限。
- [ ] 支持按任务路由模型，例如案例分析使用强模型、格式检查使用低成本模型。
- [ ] 处理供应商差异：系统消息、JSON Schema、停止条件、错误码和内容过滤。
- [ ] 建立录制/回放式假模型，单元测试不得调用真实付费 API。

验收：相同工作流可仅修改配置切换至少三类 endpoint；故障不会破坏已保存状态；预算超限时安全停止。

## 4. 文档摄入与标准化（M2）

- [ ] 实现 TXT 与 Markdown 解析，保留标题层级和段落位置。
- [ ] 实现 DOCX 解析，提取标题、正文、表格、脚注/尾注、页眉页脚和样式信息。
- [ ] 实现文字版 PDF 解析，保留页码并检测乱码、双栏错序和空文本页。
- [ ] 对疑似扫描 PDF 明确报错并说明首版不支持 OCR。
- [ ] 统一输出为内部文档树，保留原始文本与规范化文本两份表示。
- [ ] 实现编码、空白、断行、页眉页脚重复和参考文献区识别。
- [ ] 检测并隔离文档内可能针对 Agent 的 Prompt Injection 指令。
- [ ] 增加文件大小、页数、压缩包和资源消耗限制。
- [ ] 为四种格式建立无隐私的固定测试样本和解析快照测试。

验收：同一文章的四种格式能产生语义等价的章节树；每段可定位回源文件；异常 PDF 不会静默生成错误文本。

## 5. 作业要求与体裁识别（M2）

- [ ] 从任务说明、评分标准和模板中抽取硬约束与软偏好。
- [ ] 识别论文类型，并在不确定时给出候选及依据，不擅自选择。
- [ ] 提取题目范围、受众、目的、字数、章节、格式、引用、数据和截止时间。
- [ ] 实现规则优先级：教师/学校 → 用户事实 → 已确认 Skill → 内置体裁 → 默认值。
- [ ] 内置四类 Genre Profile，分别定义结构、章节目标、必需证据和质量规则。
- [ ] 支持用户交互式修正抽取结果，并保存为项目约束。
- [ ] 在生成前输出“任务理解确认单”。

验收：对测试任务书的关键硬约束召回率达到预设阈值；存在冲突时阻止生成并列出冲突来源。

## 6. 案例学习与 Writing Skill 生成（M3，核心能力）

### 6.1 案例语料准备

- [ ] 支持一次导入多份案例，并记录用户声明的优秀依据、课程、学科和成绩信息（均可选）。
- [ ] 检查案例是否属于同一任务分布；混杂体裁时先聚类或要求拆分。
- [ ] 自动脱敏姓名、学号、教师、院系、联系方式等个人信息。
- [ ] 切分标题、摘要、章节、段落、引文和参考文献，不把参考文献语言误学为正文风格。
- [ ] 对过少、质量冲突或高度重复的案例给出风险提示，不伪造统计可靠性。
- [ ] 用基准测试校准“案例足够”的门槛；初始 Demo 最低 3 篇同类型案例、建议 5 篇，不仅按数量判定。

### 6.2 特征归纳

- [ ] 提取宏观结构、章节比例和章节间依赖。
- [ ] 提取每节修辞动作，例如背景—问题—论点、概念—证据—解释、局限—建议。
- [ ] 建立段落角色序列和 Claim–Evidence–Warrant 模式。
- [ ] 提取文献综合方式、引用密度、引文位置和来源类型偏好。
- [ ] 提取语言风格统计，但禁止保存可导致原句复用的大段文本。
- [ ] 区分跨案例稳定规则、单案例偶然特征和互相冲突的做法。
- [ ] 将每条规则绑定支持案例、位置、支持度与反例。
- [ ] 将案例规则与内置学术规范比对，显式记录冲突。

### 6.3 Skill 编译、验证与版本化

- [ ] 把归纳结果编译为结构化 Skill 文件，而非仅生成 Prompt 文本。
- [ ] 由 Skill 编译器生成各阶段 Prompt 片段、验证规则和评分 Rubric。
- [ ] 实现留一案例验证：用其余案例归纳 Skill，检查对留出案例结构的解释力。
- [ ] 实现反复制测试：生成内容与案例做连续短语/相似片段检查。
- [ ] 生成 Skill 报告：高置信规则、低置信规则、冲突、样本不足和适用范围。
- [ ] 达到质量门槛时自动冻结 Skill 版本并进入规划/写作；阻断级冲突才暂停。
- [ ] 提供可选 CLI 审阅流程，用户可接受、修改、禁用规则后再冻结版本。
- [ ] 支持 Skill 的导出、导入、语义版本、升级比较和回滚。
- [ ] Skill 文件默认不嵌入案例全文，避免隐私与版权泄露。

验收：对准备好的案例集能输出可读报告和机器可执行 Skill；规则均有证据；替换案例后结果可比较；生成稿不复用案例的长片段。

## 7. 来源、证据与引用系统（M4）

- [ ] 建立来源台账，区分用户资料、已核验外部来源、候选来源和不可用来源。
- [ ] 从文档中提取作者、标题、年份、出版物、DOI/URL、页码等元数据。
- [ ] 接入公开元数据服务核验 DOI 和基础书目信息，并提供离线降级路径。
- [ ] 为每个来源生成带原文定位的 Evidence Card：主张、方法、结论、局限、可引用片段。
- [ ] 生成“来源 × 主题/方法/观点”的文献综合矩阵。
- [ ] 外部事实、数据和观点必须先绑定 Evidence Card，才能进入正式草稿。
- [ ] 不确定来源用显式占位符，不允许模型补齐看似合理的书目信息。
- [ ] 实现直接引语长度、改写过近、漏引、错引和文中/文后不一致检查。
- [ ] 支持 GB/T 7714-2025，并规划 GB/T 7714-2015 兼容模式和用户模板覆盖。
- [ ] 输出逐条引用核验报告，列出 DOI/URL/页码和未核验项。

验收：删除任一来源时，可列出受影响的句子和论点；虚构 DOI 测试必须失败；正文引用与参考文献表双向一致。

## 8. 规划 Agent：从任务到可写提纲（M4）

- [ ] 生成候选研究问题并检查清晰度、范围、可回答性和资料充分度。
- [ ] 生成可争辩、范围适当且与任务相符的中心论点候选。
- [ ] 为中心论点构建分论点、证据、推理桥梁、反论点和回应。
- [ ] 检测“只有主题没有问题”“只有观点没有证据”“证据不能推出结论”等失败。
- [ ] 按 Genre Profile 与 Writing Skill 生成带章节目标、字数预算和来源预算的提纲。
- [ ] 为每一节生成完成定义：要回答的问题、必须使用的证据和不可重复内容。
- [ ] 在证据不足时返回研究缺口清单，而不是直接进入写作。
- [ ] 提供交互式提纲审阅、局部重写、锁定节点和版本比较。

验收：提纲中每个实质章节都服务于中心问题；每个事实性分论点有证据计划；总字数预算与章节预算一致。

## 9. 写作 Agent：分层生成高完成度草稿（M5）

- [ ] 按“全文计划 → 章节计划 → 段落计划 → 段落草稿”分层生成。
- [ ] 生成时只提供当前章节所需来源和全局摘要，控制上下文污染。
- [ ] 每段保存段落角色、所支持 Claim 和所使用 Evidence ID。
- [ ] 实现通识论文的论点—证据—解释—回应写作模式。
- [ ] 实现综述的跨来源综合，禁止默认按作者逐篇罗列。
- [ ] 实现调研报告的方法/结果/讨论分离和数据真实性门禁。
- [ ] 实现本科论文的章节衔接、研究问题回扣和局限说明。
- [ ] 支持引用安全的直接引语、改写、摘要与作者分析模式。
- [ ] 支持章节级反馈与重写，锁定的内容不得被全局修订覆盖。
- [ ] 建立术语表、缩写表、人名/机构名表和关键结论表。
- [ ] 执行全局修订：消除重复、修复前后矛盾、改善过渡、统一语气和篇幅。
- [ ] 对无证据支持的内容保留醒目的待补标记，并汇总给用户。

验收：四类固定测试任务均能完成从提纲到草稿；全文不存在孤立章节；所有外部事实可追溯；锁定内容保持不变。

## 10. 质量评估与审计 Agent（M5）

- [ ] 建立确定性检查与 LLM 评审分离的评估框架。
- [ ] 检查任务满足度：题目、体裁、字数、章节、格式和评分标准。
- [ ] 检查论证质量：论点明确性、证据覆盖、推理跳跃、反论点和结论回扣。
- [ ] 检查综述质量：综合程度、分类合理性、共识/分歧/空白。
- [ ] 检查调研真实性：样本、方法、数据和结论是否对应。
- [ ] 检查引用质量：存在性、可定位性、支持关系、格式和引用完整性。
- [ ] 检查全文一致性：术语、数字、日期、缩写、章节引用和结论。
- [ ] 检查语言：病句、空泛套话、过度断言、口语化、重复和段落衔接。
- [ ] 检查与案例及来源的过近复述，输出命中位置而非承诺“查重率”。
- [ ] 使用不同模型或独立上下文进行二次评审，降低自评偏差。
- [ ] 生成分级审计报告：阻断项、重要项、建议项、需人工确认项。
- [ ] 只自动修复低风险问题；事实、论点和引用变更必须请求确认。

验收：预置的伪造文献、无证据数字、前后矛盾、综述摘要串联等故障均能被测试捕获；报告能定位到章节/段落。

## 11. 输出与格式化（M6）

- [ ] 输出 Markdown：正文、参考文献、待补项和审计报告分离。
- [ ] 输出 DOCX：标题层级、目录、分页、页码、脚注/尾注、表格和参考文献样式。
- [ ] 支持学校 DOCX 模板映射，无法识别的样式给出警告。
- [ ] 实现中文排版基础规则：中英文标点、数字/单位、标题编号和空格。
- [ ] 实现参考文献样式选择及文中引用/文后列表同步渲染。
- [ ] 将未核验引用、待补数据和用户确认项写入单独检查清单。
- [ ] 通过渲染后的 PDF/页面图进行 DOCX 视觉回归检查（仅测试输出，不把 PDF 作为首版主输出）。

验收：示例文档打开无修复提示；目录和交叉引用有效；模板测试无明显溢出、孤行标题或错乱编号。

## 12. CLI 用户体验（M6）

- [ ] 设计命令：`init`、`ingest`、`skill build`、`skill review`、`plan`、`draft`、`audit`、`export`、`status`、`resume`。
- [ ] 提供 `run` 端到端命令：案例足够且无阻断问题时自动执行 Skill → 规划 → 草稿 → 审计 → 导出。
- [ ] 提供从零开始的交互式向导，同时允许 CI/脚本使用非交互参数。
- [ ] 每个长任务显示当前阶段、预计调用量、费用和可恢复检查点。
- [ ] 对缺少 API Key、格式不支持、上下文过长和来源不足给出可执行建议。
- [ ] 提供 `--dry-run`，预览模型、预算、输入和预期产物。
- [ ] 提供 `--offline`，禁用联网核验并在报告中声明能力下降。
- [ ] 支持中文默认界面，并为未来英文界面预留 i18n。

验收：新用户仅阅读 `--help` 和快速开始即可走通示例；失败后能用 `resume` 恢复；非交互模式退出码可靠。

## 13. 测试、基准与安全（M7）

- [ ] 为解析器、Schema、规则优先级、引用渲染和状态机建立单元测试。
- [ ] 为四种论文类型各建立至少一套合成端到端夹具，不使用真实学生隐私材料。
- [ ] 建立案例 Skill 基准：结构归纳、规则稳定性、留一验证和防复制。
- [ ] 建立多模型契约测试，检查结构化输出和错误降级。
- [ ] 建立黄金草稿 Rubric：任务满足、论证、证据、综合、一致性、语言和格式。
- [ ] 同时记录人工评分与自动评分，不把单一 LLM 评分当作真值。
- [ ] 注入故障样本：假 DOI、假数据、错页码、Prompt Injection、恶意 DOCX/PDF、超大文件。
- [ ] 做隐私威胁建模：本地文件、API 发送范围、日志脱敏、临时文件和删除策略。
- [ ] 建立性能与成本基准，定义不同篇幅下的时间和预算上限。
- [ ] 设置发布门槛：关键测试全通过、阻断级审计故障召回达到目标、无真实隐私数据。

验收：固定种子/回放条件下基准可复现；安全夹具不会执行文档指令或泄露密钥；发布门槛可由 CI 自动判断。

- [ ] 编写安装、模型配置、Writing Skill、引用审计和 DOCX 模板文档。

## 建议实施顺序

1. M0–M2：做出可靠的 CLI、数据模型、模型切换和文档解析底座。
2. M3：优先验证案例到 Skill 的核心假设，未通过基准前不扩张功能。
3. M4：完成证据台账和规划 Agent，建立“不带证据不写正文”的门禁。
4. M5：实现四类写作与审计闭环，先质量后格式。
5. M6–M8：完成 DOCX、用户体验、系统测试和开源发布。

## 首个可演示纵切片

首个 Demo 不等待全部功能完成，限定为：

- 输入：3–5 篇同类型 Markdown/TXT 优秀案例、一份任务说明、若干真实来源。
- 体裁：通识课结课论文。
- 模型：两个可切换 endpoint。
- 流程：案例解析 → Skill 自动验证与冻结 → 论证图 → 提纲 → 分节草稿 → 引用/一致性审计 → Markdown 导出。
- v0.1 已在纵切片之后补齐 DOCX/文字 PDF 解析与 DOCX 导出；毕业论文长上下文优化和外部检索仍属于后续路线图。

Demo 验收标准：一条 `run` 命令可完成全流程；切换模型不改业务代码；Skill 规则有案例证据；草稿中的外部事实均有来源 ID；审计报告能阻断假引用；生成内容与案例无长片段复制。

# Paper Agent 项目定义

## 1. 项目目标

Paper Agent 是一个面向中文大学写作任务的命令行工具。它帮助用户完成从任务理解、案例学习、资料整理、结构规划到高完成度草稿生成和质量检查的全过程。

首版优先支持：

1. 通识课结课论文
2. 文献综述
3. 课程调研报告
4. 本科毕业论文

“高完成度草稿”指结构、论证、引用占位、格式和语言均经过系统检查，接近人工精修阶段；它不等于可绕过用户审阅、事实核验或学校规定而直接提交的成品。

## 2. 已确认的产品约束

- 交互形态：CLI。
- 模型能力：通过统一接口支持多个云端及本地模型，工作流不得绑定单一供应商。
- 主要语言：中文；专有名词和文献元数据保留原语言。
- 输入格式：DOCX、文字版 PDF、Markdown、TXT；首版不支持扫描版 PDF/OCR。
- 案例学习：用户上传若干优秀案例后，系统自动归纳、验证并保存可复用 Writing Skill。
- 学术边界：定位为写作辅导与草稿生成；保留来源追踪、人工确认和引用核验，禁止伪造文献、数据、访谈和调研过程。
- 工作目录：本项目仅在 `paper-agent/` 内开发，不依赖父目录中的其他个人资料。

## 3. 核心工作流

```text
初始化项目
  → 导入作业要求/评分标准
  → 导入优秀案例
  → 解析并归一化文档
  → 归纳 Writing Skill
  → 自动验证并冻结可用的 Skill 版本
  → 导入用户资料与真实来源
  → 生成研究问题、论点和提纲
  → 按章节生成草稿
  → 全文一致性修订
  → 事实、引用、相似度和格式审计
  → 导出 Markdown/DOCX 与审计报告
```

默认提供端到端自动模式：当案例数量、同质性、解析质量和 Skill 验证结果达到门槛时，系统从案例导入连续执行到草稿与审计报告输出，不把人工确认设为必经步骤。样本冲突、任务要求冲突、来源不足、疑似虚构数据等阻断级问题必须暂停并向用户说明。交互式审阅 Skill、提纲和草稿仍作为可选模式提供。

## 4. 写作理论到功能的映射

| 理论或原则 | 产品功能 |
|---|---|
| Flower–Hayes 写作过程理论：规划、转写和审阅是递归过程 | 可恢复状态机、章节反馈、局部重写与全文修订循环 |
| Genre-based pedagogy：文本受交际目的、受众与体裁惯例约束 | 作业解析、四类 Genre Profile、案例派生 Skill |
| Swales 修辞动作分析：章节由完成交际目的的 moves 组成 | 章节/段落角色识别、修辞动作序列、引言结构规划 |
| Toulmin 论证模型：Claim 需由 Ground/Evidence 经 Warrant 支撑，并处理 Qualifier/Rebuttal | 结构化论证图、反论点与推理压力测试 |
| 分析性问题导出中心论点 | 研究问题生成、范围检查、可争辩性检查 |
| 文献综述重在综合而非逐篇摘要 | 证据卡片、主题矩阵、观点聚类、分歧与空白识别 |
| 范例学习与形成性评价：范例需和质量标准、比较及反馈结合 | 多案例对比、规则证据、Rubric 编译、留一验证与可选审阅 |
| 引用必须可定位、可验证 | 来源台账、逐句引用绑定、DOI/URL/页码核验 |
| 格式服从学校或课程要求 | 模板优先级、格式规则配置、DOCX 渲染检查 |

## 5. 四类论文的默认结构

默认结构只是后备规则；课程模板、评分标准和经确认的案例 Skill 优先级更高。

### 通识课结课论文

问题背景 → 分析性问题 → 中心论点 → 分论点与证据 → 反方观点及回应 → 综合结论。

### 文献综述

范围与问题 → 检索/纳入说明 → 按主题、方法或时间组织的综合 → 共识与分歧 → 局限、趋势或空白 → 结论。

### 课程调研报告

背景与目标 → 调研问题 → 方法、样本与工具 → 结果 → 解释和局限 → 建议 → 结论。

系统不得虚构调研已经实施；没有真实数据时只能生成调研方案、问卷草案或带明确占位符的报告框架。

### 本科毕业论文

引言 → 文献综述/理论基础 → 研究问题 → 方法或材料 → 分析/结果 → 讨论 → 结论、局限与展望。

具体章节允许按人文、社科、管理等学科模板调整。

## 6. Writing Skill 的定义

Skill 不是案例原文摘要，也不是单段 Prompt，而是一份带来源、置信度和测试结果的结构化写作规范。至少包含：

- 适用体裁、学科、篇幅和任务场景
- 文档宏观结构及章节比例
- 各章节的修辞目标、必需内容与顺序
- 论证模式、证据类型和段落模式
- 文献组织与引用习惯
- 语言风格、术语、句长和衔接偏好
- 格式要求
- 禁止模式和常见失败
- 从哪些案例归纳而来，以及规则的支持度
- 与内置通用规范的冲突及处理方式
- Skill 版本、生成模型、验证集结果和自动/人工确认状态

仅达到“数量”不等于案例已经足够。自动模式的门槛同时考虑案例数量、任务同质性、解析完整度、规则一致性和留出验证表现。首个 Demo 以 3 篇同类型案例为最低输入、5 篇为建议输入；正式默认值须由 M3 基准测试校准，不把未经验证的固定数字写死为普遍规律。

## 7. 规则优先级

发生冲突时按以下顺序处理：

1. 学校、课程和教师明确要求
2. 用户明确要求及真实研究事实
3. 用户确认过的案例 Skill
4. 论文体裁的内置规范
5. 通用语言与格式默认值

任何层级都不能覆盖“不伪造来源、事实和数据”的硬约束。

## 8. 首版非目标

- 扫描件 OCR、手写识别和复杂公式识别
- 自动访问付费论文数据库或绕过访问限制
- 代替用户完成真实问卷、访谈或实验
- 承诺通过查重、AI 检测或学校审核
- 覆盖专业科研论文的实验设计和同行评审投稿全流程
- 图形界面、多人协作和云端账号系统
